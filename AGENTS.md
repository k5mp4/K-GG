# K-GG repository instructions

このリポジトリでは、ドキュメントを実装とテストの一次情報として扱う。

## 作業を始める前に

1. `docs/development/index.md` から開発者向け資料を読む。
2. 変更対象に対応する `docs/specs/` の仕様を特定する。
3. 関連する `docs/adr/` と仕様の `depends_on`、`related_adrs` を確認する。
4. 該当仕様がなければ、`docs/development/docdd.md` の変更区分に従って先に作成する。

## 実装ルール

- 新機能または外部から観測できる振る舞いの変更は、承認済み仕様なしに実装しない。
- 仕様とコードが矛盾した場合、独断でどちらかへ合わせない。まず差異を報告し、仕様を更新して承認状態を確認する。
- `status: approved` の仕様を実装し始めたら、完了時に `status: implemented`、`updated`、`related_code`、`related_tests` を更新する。
- アーキテクチャ、依存関係、永続化形式、配布方式に長期的な影響がある判断は ADR に残す。
- スコープ外の改善を同時に実装しない。必要なら別の仕様候補として記録する。
- バグ修正では、期待動作を既存仕様へ追記する。対応する仕様がなければ短い仕様を作る。

## サブエージェント運用

`.agents/*.yaml` は親エージェントの判断を置き換えるものではなく、実装、テスト、セキュリティレビューを分担する補助役として使う。

- `implementation-expert`: 承認済み仕様、関連ADR、対象ファイル、受け入れ条件、禁止スコープを渡してから、指定範囲の実装だけを任せる。
- `test-automation-engineer`: 仕様の受け入れ条件と実装差分を渡し、VitestまたはRustテスト、必要な手動確認観点を任せる。
- `security-auditor`: Tauri/Rust、ファイル操作、外部FFmpeg、更新機構、依存関係、外部入力を触る差分で必ずレビューに使う。

親エージェントは、サブエージェントへ渡す前に関連仕様とADRを特定し、結果を受け取った後に文書、コード、テスト、検証結果の整合を最終確認する。

## 完了条件

- 仕様の受け入れ条件をテストまたは明記した手動確認で検証している。
- `npm run docs:check` と `npm run docs:build` が成功する。
- コード変更時は変更範囲に応じて `npm test`、`npm run lint`、`npm run build` を実行する。
- コード、テスト、仕様、ADR、利用者向け説明に矛盾がない。

テンプレートや詳しいフローは `docs/development/docdd.md` を参照する。
