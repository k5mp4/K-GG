# ADR-0007: プリセットライブラリの仮想フォルダとZIP交換形式

## コンテキスト

プリセットを階層化すると、ブラウザの`localStorage`、Tauriのアプリデータ、外部共有ファイルの三つで同じ構造を表現する必要がある。ブラウザのファイルAPIやTauriのOSフォルダAPIに依存すると、実行環境ごとに操作と権限が変わる。また、既存の保存データは単純な`Preset[]`である。

## 決定

- プリセットの階層はOS上の実フォルダではなく、バージョン付きライブラリ文書の仮想フォルダとして保持する。
- ルートフォルダは`null`の親IDで表し、フォルダとプリセットはID・親ID・表示順を持つ。
- 単一プリセットは既存JSON互換で扱い、フォルダ/ライブラリ全体は既存依存の`fflate`でZIP化する。
- ブラウザとTauriは同じ正規化・検証・アーカイブ処理を共有し、保存先の読み書きだけを各アダプターに残す。
- インポートは指定フォルダへの追加とし、IDを再採番する。既存データの置換は行わない。

## 理由

仮想フォルダならブラウザとTauriの挙動を一致させられ、OS権限やファイルパスの違いをUIへ持ち込まずに済む。ZIPは階層を一つのファイルで交換でき、既存依存を再利用できる。旧配列を正規化することで配布済みアプリの保存データも移行できる。

## 代替案

| 案 | 採用しなかった理由 |
| --- | --- |
| TauriではOSの実フォルダを直接管理する | ブラウザ版との機能差と権限差が大きくなる。 |
| プリセット名に`カテゴリ/サブカテゴリ`を埋め込む | 空フォルダや名称変更を表現できず、表示順も保持できない。 |
| フォルダごとJSONを複数ファイルで保存する | ブラウザの永続化と共有ファイルの扱いが複雑になる。 |

## 結果

### 利点

- 実行環境に依存しない階層管理と共有ができる。
- 既存データを段階的に移行できる。
- 既存のZIP依存を再利用できる。

### 欠点・コスト

- OSのファイルブラウザから直接フォルダを編集する用途には対応しない。
- ライブラリ文書のバージョン管理とインポート検証が必要になる。
- ZIP展開時のサイズ上限や不正エントリ検証を実装する必要がある。

## 再検討条件

- 利用者がOSフォルダとの同期を必要とし、Tauri専用機能として明確な価値が生まれた場合は再評価する。
- プリセット以外のアセットを同じライブラリで管理する必要が生じた場合は、アーカイブ形式のバージョンを再検討する。
