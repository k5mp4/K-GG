# ADR-0006: Tweeqを共通入力コントロールとして採用する

## コンテキスト

K-GGでは、数値パラメータのドラッグ・直接入力と色の選択UIを複数のコンポーネントで実装している。指定された`tweeq`はクリエイティブツール向けのReactコンポーネント群で、`InputNumber`と`InputColor`を提供する。一方、GitHubリポジトリの現在のReact版は生成済み`lib`をリポジトリへ含めず、`package.json`も開発用のソースリポジトリとして構成されている。

主要ライブラリの採用と、ライブラリのソース解決方法は将来の入力UIと依存更新を拘束するため、機能仕様とは別に判断を記録する。

## 決定

- K-GGの共通数値入力と色入力にTweeqのReactコンポーネントを採用する。
- 依存先は`arcatdmz/tweeq`の`d859beddb0228b56d7ea5dbd1a5f904f559b81fa`へ固定する。
- 生成済み`lib`が存在しないGitHub版をそのまま依存入口から読むのではなく、指定コミットから生成した最小の配布物を`vendor/tweeq`へ固定する。生成物にはコミットIDとMITライセンスを併記する。
- K-GG側にTweeqの入力コンポーネントを直接複製せず、既存の`SliderField`と`ColorPicker`を薄い互換アダプターとして残す。

## 理由

指定リポジトリの現行APIを使うにはReact版を採用する必要がある。コミット固定により、外部リポジトリの更新で操作やスタイルが予告なく変わることを防げる。生成物をリポジトリへ含めることで、利用者の環境でGitHub取得やライブラリの再ビルドを必須にせず、通常の`npm ci`で再現できる。アダプターを残すことで、Zustand、キーフレーム、デフォルト値、既存の表示ラベルの契約をTweeqのAPIから分離できる。

## 代替案

| 案 | 採用しなかった理由 |
| --- | --- |
| npm公開版`tweeq`を使用する | 調査時点の公開版はREADMEとAPIが古いVue系のため、指定GitHubのReact版と一致しない。 |
| TweeqのソースをK-GGへコピーする | 上流との同期が難しく、著作権表示・差分管理・更新コストが増える。 |
| 既存の`react-colorful`と独自`SliderField`を継続する | 今回の共通操作モデル導入という目的を満たさない。 |

## 結果

### 利点

- 数値入力と色入力で、ドラッグ・直接編集・微調整の操作モデルを共通化できる。
- 上流の入力コンポーネントと型を利用でき、K-GGのUI実装量を抑えられる。
- アダプター境界により、K-GGの状態・描画層へTweeqのストアを漏らさずに済む。

### 欠点・コスト

- 外部GitHubリポジトリのソース解決と、Tweeq固有の依存・Stylusビルド設定をK-GGで維持する必要がある。
- TweeqのCSSリセットやCSS変数がK-GGの作業領域と競合しないよう、Viewport配下を確認する必要がある。
- 依存追加でインストール時間と配布バンドルサイズが増える。

## 再検討条件

- Tweeqが安定した公開npmパッケージとしてReact版の生成物と型を提供した場合は、GitHubソースエイリアスからnpm依存へ移行を検討する。
- Tweeqのメンテナンス停止、ライセンス変更、またはK-GGの操作契約との非互換が発生した場合は採用を再評価する。
