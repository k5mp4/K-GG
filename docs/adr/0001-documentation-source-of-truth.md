# ADR-0001: リポジトリ内文書を開発の一次情報とする

## コンテキスト

K-GGはReact、WebGL/GLSL、Tauri/Rust、複数の出力経路を持ち、変更がUI、状態、描画、保存、配布へ波及する。コードだけでは「なぜその動作にしたか」や対象外を判断しにくい。AI支援開発では、自然言語の依頼ごとに解釈が変わると、動くが意図と異なる実装を高速に増やす危険がある。

## 決定

機能の意図と期待動作は、リポジトリ内のバージョン管理された仕様書を一次情報とする。コードとテストは仕様の実現物として同じPull Requestで同期する。

機能仕様は`docs/specs/`、長期的な技術判断は`docs/adr/`、横断的な現状説明は`docs/development/`に分離する。新規機能と観測可能な動作変更は、人間が仕様を承認してから実装する。

## 理由

- コード、仕様、レビューを同じ履歴と変更単位で管理できる。
- 人間と異なるAIツールへ同じコンテキストを渡せる。
- 実装前に要求の曖昧さと設計上の手戻りを発見できる。
- 受け入れ条件からテストへ追跡できる。

## 代替案

| 案 | 採用しなかった理由 |
| --- | --- |
| コードとコメントだけを正とする | 背景、対象外、代替案を保ちにくい |
| Issueやチャットだけで仕様を管理する | リポジトリの版と対応せず、AIや新規参加者が発見しにくい |
| 外部Wikiを一次情報にする | コードレビューと同期した更新を強制しにくい |
| 特定AIツールの設定だけに規約を書く | 開発者と他のツールから参照できない |

## 結果

### 利点

- 実装の判断根拠と変更履歴が残る。
- Pull Requestで仕様逸脱をレビューできる。
- 新しい開発者とAIのオンボーディングが共通化される。

### 欠点・コスト

- 観測可能な変更では実装前後に文書を更新する必要がある。
- 古い文書を放置すると誤誘導が強くなるため、同期を完了条件に含める必要がある。
- 仕様変更の最終判断は自動化できず、人間の承認が必要である。

## 再検討条件

仕様とコードの対応を継続的に維持できない、または文書運用が変更リードタイムを不合理に増やしていると定量的に確認された場合、テンプレートと変更区分を見直す。一次情報を外部へ移す場合も、コードの版と厳密に対応できることを条件とする。
