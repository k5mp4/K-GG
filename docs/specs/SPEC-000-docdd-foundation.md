# SPEC-000: DocDD基盤の導入

## 背景・問題

K-GGには利用者向けガイドとビルド手順がある一方、新しい開発者がシステム構造、変更理由、仕様とコードの関係を追うための入口が不足していた。AI支援開発でも、セッションごとに前提が変わると、要求のずれや意図しない追加実装が発生する。

## ゴール・成功条件

- 新しい開発者が文書の入口から目的、構造、検証方法を把握できる。
- 機能変更のWhy、What、How、受け入れ条件を実装前にレビューできる。
- AIコーディングエージェントが同じ仕様と作業ルールを参照できる。
- ユーザーが公開を依頼した変更では、AIコーディングエージェントが日本語の説明を持つドラフトPull Requestまで作成できる。
- 仕様とADRの最低限のメタデータを自動検証できる。

## スコープ

### 対象

- 開発者向け概要、アーキテクチャ、開発手順
- 機能仕様とADRのディレクトリ、テンプレート、索引
- 変更規模に応じたDocDDワークフロー
- AIエージェント向けリポジトリ指示
- Pull Requestでの仕様トレーサビリティ
- 文書メタデータとVitePressビルドの検証
- CIでの文書メタデータ検証と文書サイトビルド

### 対象外

- 既存機能すべての詳細仕様を遡って作成すること
- 特定のAI製品、MCP、エディターを必須にすること
- アプリケーションの利用者向け機能を変更すること

## 方針

利用者向け文書、開発者向け恒常文書、機能単位の仕様、長期的な意思決定を分離する。仕様には機械可読なフロントマターを持たせ、ID、状態、依存、関連コード、関連テスト、人間レビュー状態を記録する。

既存機能の完全な遡及文書化は行わない。プロジェクト全体の地図を先に提供し、新規機能と変更対象から仕様を蓄積する。これにより、導入時の文書化コストを制限しながら、今後の変更を追跡可能にする。

## 代替案

| 案 | 採否 | 理由 |
| --- | --- | --- |
| すべてを単一READMEへ記載 | 不採用 | 読者と更新周期が異なり、変更時に追跡しにくい |
| 外部Wikiのみで管理 | 不採用 | コードと同じ変更履歴・レビュー単位を保てない |
| 全変更に完全な仕様を必須化 | 不採用 | 軽微な変更のコストが高く、運用が形骸化する |
| 変更区分に応じて文書量を変える | 採用 | 小さく開始しつつ観測可能な変更を確実に仕様化できる |

## エラー・境界条件

- 仕様IDの重複、不正な状態、存在しない依存IDは検証を失敗させる。
- テンプレートは未確定のプレースホルダーを持つため検証対象外とする。
- 文書と実装が異なる場合は、自動的に片方を正とせず人間の判断へ戻す。
- 文書のみの変更には、アプリケーション全体の仕様書作成を要求しない。

## 受け入れ条件

- AC-001: 開発者向け入口から概要、アーキテクチャ、開発手順、DocDD、仕様、ADRへ移動できる。
- AC-002: 新規仕様とADRを一貫したテンプレートから作成できる。
- AC-003: AIエージェント向け指示が、実装前の仕様確認と完了時の同期を要求する。
- AC-004: メタデータ検証が仕様・ADRのID、状態、必須項目、依存参照を確認する。
- AC-005: Pull Requestテンプレートで仕様IDと受け入れ条件の検証結果を確認できる。
- AC-006: VitePressが新しい文書構成を内部リンク切れなしでビルドできる。
- AC-007: CIが文書メタデータ検証とVitePressビルドを実行する。
- AC-008: 公開を依頼された変更では、AIコーディングエージェントが対象差分を確認してコミット・pushし、指定がなければ日本語のタイトルと本文を持つドラフトPull Requestを作成する。

## 移行・互換性

アプリケーションの実行時データと利用者操作への影響はない。既存の`docs/index.md`を利用者ガイドとして維持し、開発者向け文書を追加する。
